# 大学生竞赛智能咨询系统并行开发计划

## 1. 计划目标

这份计划用于指导当前项目后续开发，核心原则是：

- 毕业设计答辩演示优先
- 先打通最小可演示闭环
- 在不增加过多返工的前提下，把工作拆成可并行推进的分支

当前计划严格基于现有仓库状态制定：

- 已有需求文档和 PRD
- 已有 5 个静态页面原型
- Git 仓库已建立并已连接远程
- 页面、文档目录已整理，但当前整理结果还未形成一次独立提交

## 2. 当前进度评估

### 2.1 静态原型完成度

预计完成度：`60%`

判断依据：

- 页面原型已经具备
- PRD 和需求摘要已经明确
- 页面结构已基本对应业务模块
- 项目目录和仓库基础已搭好

仍未完成：

- 页面内部导航未打通
- 页面缺少统一入口
- 页面仍是静态展示，未接真实数据

### 2.2 可演示系统完成度

预计完成度：`25% ~ 30%`

判断依据：

- 还没有数据库
- 还没有后端项目骨架
- 还没有竞赛检索和详情接口
- 还没有 AI 咨询和 SSE
- 还没有后台 CRUD
- 还没有记录与统计接口

结论：

当前已经完成“需求 + 原型 + 仓库准备”，但真正的“系统能力”还没有落地。

## 3. 当前阶段必须先补的一步

在正式拆并行分支之前，建议先做一个基线提交：

- 把当前目录重构结果提交一次
- 目标是把 `docs/`、`pages/`、`README.md` 当前状态固定下来
- 这样后续每个功能分支都从同一个干净基线切出

建议分支名：

- `chore/repo-structure-baseline`

建议产出：

- 目录重构提交
- README 更新提交
- 当前 `docs/` 和 `pages/` 结构固定

这一步应视为所有后续并行工作的前置条件。

## 4. 剩余任务拆解

### 任务包 A：前端页面联动与导航

目标：

- 打通 5 个页面的跳转关系
- 统一顶部导航和主要按钮入口
- 补一个最低成本的登录入口策略

产出物：

- 页面间可点击跳转
- 首页到详情页、详情到咨询页、统计页和后台页的可演示链路
- 如果暂时不单做登录页，则明确“演示入口方案”

依赖：

- 依赖基线提交

并行性：

- 可最早开始
- 不依赖数据库和后端

建议分支：

- `feat/frontend-routing-shell`

### 任务包 B：数据库结构与初始化数据

目标：

- 完成一期数据库表结构
- 准备演示数据

产出物：

- 建表 SQL
- 初始化分类和竞赛数据
- 记录与统计相关基础表

依赖：

- 依赖基线提交

并行性：

- 可最早开始
- 与页面联动、后端骨架可并行

建议分支：

- `feat/db-schema-seed`

### 任务包 C：后端基础骨架与竞赛接口

目标：

- 搭建后端最小可运行骨架
- 打通竞赛列表和详情接口

产出物：

- 后端目录结构
- 基础路由
- 竞赛列表接口
- 竞赛详情接口

依赖：

- 依赖基线提交
- 最终落地依赖数据库结构

并行性：

- 可与 A、B 并行启动
- 但正式联调依赖 B

建议分支：

- `feat/backend-bootstrap-competition`

### 任务包 D：智能咨询与 SSE

目标：

- 实现普通问答
- 实现流式问答
- 支持最小连续对话

产出物：

- AI 咨询接口
- SSE 流式返回接口
- 咨询记录落库

依赖：

- 依赖后端骨架
- 依赖数据库表
- 最好已有基础竞赛数据

并行性：

- 不适合过早独立推进到底
- 可在后端骨架明确后开始

建议分支：

- `feat/ai-consult-sse`

### 任务包 E：后台管理与记录统计

目标：

- 实现分类和竞赛信息 CRUD
- 实现记录与统计接口

产出物：

- 分类 CRUD
- 竞赛信息 CRUD
- 检索记录、咨询记录、导航记录查询
- 热门统计接口

依赖：

- 依赖数据库结构
- 依赖后端骨架

并行性：

- 可在 C 和 B 稳定后并行推进
- 可与 D 并行

建议分支：

- `feat/admin-crud-statistics`

### 任务包 F：集成联调与答辩演示优化

目标：

- 把前端原型和后端能力真正接起来
- 保证一条从检索到咨询到记录的完整演示链路

产出物：

- 前后端联调完成
- 演示数据可跑通
- 页面交互与提示优化

依赖：

- 依赖 A、C
- AI 咨询依赖 D
- 后台和统计依赖 E

并行性：

- 不适合最早开始
- 必须放在中后期作为收口分支

建议分支：

- `feat/integration-demo-polish`

## 5. 并行分支设计

## 5.1 分支总览

推荐分支拆分如下：

- `chore/repo-structure-baseline`
- `feat/frontend-routing-shell`
- `feat/db-schema-seed`
- `feat/backend-bootstrap-competition`
- `feat/ai-consult-sse`
- `feat/admin-crud-statistics`
- `feat/integration-demo-polish`

## 5.2 哪些可以同时开始

基线提交完成后，这三个分支可以同时开始：

- `feat/frontend-routing-shell`
- `feat/db-schema-seed`
- `feat/backend-bootstrap-competition`

原因：

- 这三项分别对应前端壳层、数据设计、后端骨架，耦合度相对低

## 5.3 哪些必须等依赖

以下分支不建议过早启动到深水区：

- `feat/ai-consult-sse`
  - 需要后端基础路由和数据库表先稳定

- `feat/admin-crud-statistics`
  - 需要数据库结构和基础后端先落地

- `feat/integration-demo-polish`
  - 必须等前面主要能力完成后再做收口

## 6. 推荐执行顺序

### 6.1 必须先做

1. `chore/repo-structure-baseline`
2. `feat/frontend-routing-shell`
3. `feat/db-schema-seed`
4. `feat/backend-bootstrap-competition`

说明：

- 这几项决定后面所有工作是否能稳定展开
- 尤其是数据库结构和后端骨架，不能拖到很后面

### 6.2 可并行做

在上面完成基础后，可并行推进：

1. `feat/ai-consult-sse`
2. `feat/admin-crud-statistics`

说明：

- 一个偏 AI 和流式输出
- 一个偏后台和统计
- 两边共享底层表结构和后端基础，但业务代码耦合没那么高

### 6.3 后续再做

最后统一收口：

1. `feat/integration-demo-polish`

说明：

- 这个阶段主要不是“开发新功能”
- 而是把已有能力接通、清理交互细节、做答辩演示优化

## 7. 里程碑设计

### M0：仓库基线固定

完成标准：

- 当前目录重构结果提交
- 后续所有分支从统一基线切出

### M1：页面跳转打通

完成标准：

- 5 个页面之间有清晰导航
- 首页能进入详情页
- 详情页能进入咨询页
- 可进入记录统计页和后台页

### M2：数据库和演示数据可用

完成标准：

- 完成 SQL 建表
- 完成初始化分类和竞赛数据
- 能支持前台检索和后台维护的最小数据集

### M3：竞赛检索和详情链路打通

完成标准：

- 首页筛选能返回真实数据
- 点击列表能查看真实详情
- 检索记录能写入

### M4：智能咨询和 SSE 打通

完成标准：

- 普通问答可返回
- SSE 流式问答可返回
- 至少支持一轮连续追问
- 咨询记录落库

### M5：后台 CRUD 打通

完成标准：

- 分类可增删改查
- 竞赛信息可增删改查
- 后台新增的数据可在前台检索到

### M6：记录统计与答辩链路完成

完成标准：

- 咨询记录、检索记录、导航记录可展示
- 热门统计可展示
- 从后台录入到前台检索再到咨询和记录查看形成完整演示闭环

## 9. 推荐合并顺序

建议合并顺序如下：

1. `chore/repo-structure-baseline`
2. `feat/db-schema-seed`
3. `feat/backend-bootstrap-competition`
4. `feat/frontend-routing-shell`
5. `feat/ai-consult-sse`
6. `feat/admin-crud-statistics`
7. `feat/integration-demo-polish`

原因：

- 先固定仓库和目录
- 再固定数据结构
- 再固定后端骨架
- 页面联动和前端壳层尽量不要过早依赖后端细节
- AI / 后台 / 统计在中期分别推进
- 最后统一做集成和演示优化

## 10. 哪些地方适合并行，哪些不适合

### 适合并行

- 页面联动 与 数据库设计
- 数据库设计 与 后端骨架
- AI 咨询 与 后台 CRUD

### 不适合过早并行

- 不建议在数据库结构没定前就大量写后台 CRUD
- 不建议在后端骨架没定前就深做 SSE
- 不建议一开始就做统计页面真数据联调
- 不建议过早做复杂自然语言导航扩展

## 11. 风险点

### 风险 1：仓库基线不干净

当前目录重构结果还没提交，如果直接继续切功能分支，后面很容易混在一起。

### 风险 2：登录页缺口

PRD 是 6 页结构，但仓库当前只有 5 页原型。答辩时可以先采用以下策略之一：

- 用首页作为学生端默认入口，后台页作为管理员入口
- 或后续补一个极简登录页

### 风险 3：AI 和 SSE 接口不稳定

如果模型接口或流式输出不稳定，会直接影响答辩体验。

降级方案：

- 保留普通问答为主演示
- SSE 如果不稳定，可预留静态流式演示或本地模拟流

### 风险 4：统计功能过度复杂

统计很容易越做越重。

降级方案：

- 第一版先做列表 / 卡片式统计
- 不强制上复杂图表

## 12. 最小可演示版本定义

答辩时最少必须跑通的能力建议是：

1. 后台新增一个竞赛分类和竞赛信息
2. 前台首页检索到该竞赛
3. 点击进入详情页查看完整信息
4. 从详情页进入智能咨询页
5. 发起一次普通问答
6. 发起一次 SSE 流式问答
7. 在记录与统计页看到咨询记录和热门统计

## 13. 可降级功能

如果时间不够，可以降级的内容如下：

- 登录页：先不单独做，改为直接从首页和后台页分别演示
- 自然语言导航：先只做两条规则指令，不做复杂语义识别
- 连续对话：先支持最小一轮上下文，不做完整多轮会话管理
- 统计图表：先用文本列表和卡片，不强求图表
- SSE：若真实流式风险大，可准备 mock 演示兜底

## 14. 单人开发时的最优顺序

如果最后实际上主要还是你一个人推进，建议按下面顺序压缩执行：

1. 基线提交
2. 页面跳转打通
3. 数据库建表和初始化数据
4. 后端骨架 + 竞赛列表 / 详情接口
5. 前台检索和详情联调
6. AI 咨询普通问答
7. SSE 流式问答
8. 后台分类和竞赛 CRUD
9. 记录与统计
10. 答辩演示优化

这个顺序最稳，因为它优先保证一条最短闭环先跑通。
