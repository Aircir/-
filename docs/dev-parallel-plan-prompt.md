# 并行开发计划 Prompt

这份文件用于把当前项目的真实状态交给另一个 AI，让它帮助你设计“可并行执行”的开发顺序、分支划分和合并方案。

## 推荐用法

把下面这段“快捷发送模板”连同后面的完整版 Prompt 一起复制给目标 AI。

```text
请你作为技术负责人，基于下面这个 prompt，直接输出一份可执行的并行开发方案。
要求：
- 结合已有 5 个页面
- 优先保证毕业设计答辩演示
- 给出具体分支名、任务内容、依赖关系、合并顺序
- 输出尽量具体，不要空泛

【下面开始是项目 prompt】
```

## 主控 Prompt

```text
你现在是一个资深技术负责人（Tech Lead）+ 项目经理（PM）+ 架构师，请基于下面这个项目的真实现状，设计一份“适合并行执行”的开发方案。

你的目标不是泛泛提建议，而是输出一份真的可以拿来执行的开发计划，尤其要适合：
1. 毕业设计答辩演示优先
2. 当前已有静态页面原型
3. 希望拆多个 Git 分支并行推进
4. 后续可能由多个 AI / 多个开发者同时处理不同模块

【项目名称】
大学生竞赛智能咨询系统

【项目目标】
- 第一目标：毕业设计答辩演示
- 第二目标：尽量做成可实际运行的系统
- 终端：电脑网页端
- 用户角色：学生、管理员

【当前项目状态】
仓库当前已经完成以下内容：
1. 需求和产品文档已完成
- docs/PRD.md
- docs/requirements-summary.md

2. 页面原型已完成，目前有 5 个页面
- pages/home.html：首页 / 竞赛检索页
- pages/competition-detail.html：竞赛详情页
- pages/ai-consult.html：智能咨询页
- pages/records-statistics.html：记录与统计页
- pages/admin-competition-management.html：后台管理页

3. 当前仓库已经完成
- 本地 Git 仓库建立
- 远程仓库已连接
- 页面文件和文档结构已整理
- 页面目前主要是静态 HTML 原型
- 页面之间很多导航还没真正接通
- 还没有真实数据库、后端接口、AI 对接、SSE、后台 CRUD

【一期功能范围】
1. 首页 / 竞赛检索
- 关键词搜索
- 筛选条件：专业、赛事级别、报名状态
- 可保留类别、年级作为增强项
- 竞赛列表展示
- 进入竞赛详情页

2. 竞赛详情
- 展示竞赛名称、类别、级别、适合专业、适合年级、报名时间、官网、简介、参赛要求
- 提供“去智能咨询”入口

3. 智能咨询
- 支持普通问答
- 支持 SSE 流式问答
- 支持连续对话
- 可基于竞赛事实信息回答
- 可补充少量通用建议

4. 自然语言导航
一期只先支持两条指令：
- 打开竞赛列表页
- 搜索人工智能类竞赛

5. 记录与统计
- 咨询记录
- 检索记录
- 导航记录
- 全局热门统计

6. 后台管理
- 竞赛分类增删改查
- 竞赛信息增删改查

【技术现实】
- 前端已经有 HTML 原型，要以现有页面为基础改造
- 目前没有真实后端
- 后端后续会接数据库、AI 接口和 SSE
- 希望优先打通最小可演示链路
- 希望拆多个 Git 分支并行推进
- 输出要贴合当前已有 5 个页面，不要脱离现状空谈

【你的任务】
请输出一份“并行开发执行计划”，必须包含以下内容：

1. 当前进度评估
- 给出整体完成度百分比
- 分别从“静态原型完成度”和“可演示系统完成度”两个维度估算
- 说明判断依据

2. 剩余任务拆解
- 按模块拆成任务包
- 每个任务包写清目标、产出物、依赖项
- 标明哪些任务可以并行，哪些必须串行

3. Git 分支设计
- 为每个任务包设计建议分支名
- 说明每个分支负责什么
- 说明哪些分支可以同时开始
- 说明哪些分支必须等待其他分支完成后再进行
- 说明推荐的合并顺序

4. 推荐执行顺序
请分成三层：
- 必须先做
- 可并行做
- 后续再做
重点优先保证“答辩演示版最短路径”

5. 里程碑设计
至少给出这些里程碑：
- M1：页面跳转打通
- M2：数据库和演示数据可用
- M3：竞赛检索和详情链路打通
- M4：智能咨询和 SSE 打通
- M5：后台 CRUD 打通
- M6：记录统计与答辩演示链路完成

6. 并行协作建议
假设后续会由多个开发者或多个 AI 并行工作，请为每个分支写：
- 负责人类型（前端 / 后端 / 数据 / AI / 集成）
- 输入依赖
- 输出结果
- 合并前自测内容
- 与其他分支的交接点

7. 风险点
- 哪些地方容易卡住
- 哪些模块不适合过早并行
- 哪些功能可以做“演示降级方案”

8. 最终最小可演示版本定义
请明确答辩时最少必须跑通哪些功能
并指出哪些内容如果时间不够可以降级成静态展示或半真实数据

【输出要求】
- 输出必须清晰、结构化、具体
- 不要只给大而化之的项目管理建议
- 一定要结合当前已有 5 个 HTML 页面和一期需求范围
- 一定要突出“毕业设计答辩优先”
- 一定要指出并行边界和依赖关系
- 输出风格要像真实项目排期文档
```

## 可选追加问题

如果你拿到第一版输出后，还想让它更贴合自己当前阶段，可以继续追问：

```text
如果现在只有一个人开发，请把上面的并行方案压缩成单人最高效推进顺序。
```

```text
请把上面的分支计划进一步细化成 7 天开发排期表。
```

```text
请把上面的计划改写成多个 AI Agent 的任务分配单。
```
