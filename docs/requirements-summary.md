# 大学生竞赛智能咨询系统需求梳理

## 1. 系统定位

本系统面向大学生竞赛信息服务场景，目标是把分散的竞赛信息检索、智能问答、自然语言导航和使用数据沉淀整合到一个 Web 系统中。

论文题目为“基于 C++ HTTP 服务器的大学生竞赛智能咨询系统设计与实现”，整体采用 B/S 架构：

- 前端：HTML、CSS、JavaScript
- 后端：C++17 自定义 HTTP 服务器
- 数据库：MySQL 8.0
- AI 能力：兼容 OpenAI API 的模型接口
- 流式交互：SSE

## 2. 角色划分

### 2.1 普通用户（学生）

主要诉求：

- 查找竞赛
- 查看竞赛详情
- 获取个性化咨询与推荐
- 用自然语言快速打开页面或触发检索
- 查看自己的咨询记录

### 2.2 管理员

主要诉求：

- 维护竞赛分类
- 维护竞赛信息
- 管理用户账号
- 查看咨询记录、检索记录、导航记录
- 查看基础统计数据

## 3. 核心功能

### 3.1 竞赛检索

用户可以：

- 浏览竞赛列表
- 按关键词检索
- 按竞赛类别筛选
- 按适合专业筛选
- 按适合年级筛选
- 按赛事级别筛选
- 按报名状态筛选
- 查看竞赛详情

详情页建议展示：

- 竞赛名称
- 竞赛类别
- 竞赛级别
- 适合专业
- 适合年级
- 报名开始时间
- 报名结束时间
- 官网链接
- 竞赛简介
- 参赛要求

### 3.2 智能咨询

系统支持围绕竞赛信息的自然语言问答，典型问题包括：

- 我适合参加哪些竞赛
- 某个竞赛需要准备什么
- 两个竞赛有什么区别

能力要求：

- 问题预处理
- 关键词提取
- 问题类型识别
- 从数据库检索相关竞赛事实信息
- 构造提示词上下文
- 调用大模型生成回答
- 保存咨询记录

回答模式：

- 普通问答：一次性返回 JSON
- 流式问答：通过 SSE 增量返回

### 3.3 自然语言导航

用户可直接输入操作型指令，例如：

- 打开竞赛列表页
- 搜索人工智能类竞赛
- 查看最近的咨询记录

系统需要完成：

- 输入预处理
- 规则匹配
- 复杂场景下调用模型识别意图
- 参数提取
- 返回前端可执行的结构化 JSON 指令

输出结果建议至少包括：

- `action`
- `target_page`
- `keyword`
- `params`

### 3.4 咨询记录与统计管理

系统需要保存三类行为数据：

- 检索记录
- 咨询记录
- 导航指令记录

统计内容包括：

- 咨询总次数
- 热门问题关键词
- 热门竞赛类别
- 常用导航指令
- 近期活跃时间段

## 4. 支撑功能

论文中明确出现，但不是主线展示的支撑能力包括：

- 用户注册
- 用户登录
- 用户身份区分（学生 / 管理员）
- 用户状态管理
- 后台竞赛分类管理
- 后台竞赛信息管理
- 用户账户管理

## 5. 数据实体

论文中已经明确的主要数据表：

- `user_info`
- `competition_category`
- `competition_info`
- `search_log`
- `consult_record`
- `navigation_command`
- `statistic_summary`

实体关系大致为：

- 一个竞赛分类对应多个竞赛
- 一个用户可以产生多条检索记录
- 一个用户可以产生多条咨询记录
- 一个用户可以产生多条导航记录

## 6. 当前论文里的需求边界

### 6.1 明确属于必须做的

- 用户端竞赛检索
- 用户端竞赛详情查看
- 智能咨询
- SSE 流式输出
- 自然语言导航
- 咨询记录保存
- 检索/咨询/导航统计
- 管理员维护竞赛分类和竞赛信息

### 6.2 提到但展开不够，建议二选一明确

- 收藏竞赛
- 用户个人历史记录页面的完整交互
- 管理员对用户的增删改查范围
- 统计页面是否只看列表，还是要图表展示

说明：

论文在需求描述中提到“收藏竞赛”，但数据库设计与后续核心实现章节没有对应收藏表，也没有单独功能实现说明。因此这个功能当前更像“可选扩展需求”，不是论文主线刚需。

## 10. 已确认的一期需求

根据当前沟通结果，第一版系统按以下边界实现：

### 10.1 项目目标

- 以毕业设计答辩演示为主
- 同时尽量保证系统可实际使用

### 10.2 角色范围

- 仅保留学生用户
- 仅保留管理员用户
- 不引入教师角色

### 10.3 竞赛数据来源

- 由管理员在后台维护竞赛数据
- 第一版按后台手动录入或手动维护理解
- 暂不把 Excel/CSV 批量导入作为一期刚需

### 10.4 检索条件

当前明确必须支持的筛选条件：

- 专业
- 赛事级别
- 报名状态

建议保留但可选的条件：

- 关键词
- 竞赛类别
- 年级

### 10.5 智能咨询能力

- 支持围绕竞赛信息进行问答
- 可以提供一定的通用建议
- 支持连续对话
- 支持普通问答
- 支持 SSE 流式问答

### 10.6 自然语言导航一期范围

第一版优先支持以下高频指令：

- 打开竞赛列表页
- 搜索人工智能类竞赛

### 10.7 记录与统计展示范围

- 管理员可以查看全站统计
- 学生也可以查看全局热门统计
- 学生仍以查看自己的历史记录为主

### 10.8 后台一期范围

第一版后台以基础数据维护为核心：

- 竞赛分类增删改查
- 竞赛信息增删改查

暂不把以下内容作为一期刚需：

- 复杂用户管理
- 用户封禁流程
- 高级记录筛选
- 批量导入

### 10.9 终端与前端协作方式

- 系统以电脑网页端为主
- 前端页面初稿由其他工具生成
- 后续在现有前端基础上调整和接入后端功能

## 11. 精简后页面建议

结合已确认范围，第一版建议收缩为 5 到 6 个页面。

### 方案 A：5 页版本（更适合答辩）

- 登录页
- 首页 / 竞赛检索页
- 竞赛详情页
- 智能咨询页
- 后台管理页

说明：

- 记录与统计可先作为首页下方分区或单独标签区域
- 不单独做注册页，账号可由系统预置或后台维护

### 方案 B：6 页版本（更完整）

- 登录页
- 首页 / 竞赛检索页
- 竞赛详情页
- 智能咨询页
- 记录与统计页
- 后台管理页

说明：

- 该方案更利于展示“记录沉淀 + 热门统计”能力
- 如果论文答辩需要单独演示统计模块，优先选这一版
